## 八、checkpoint / resume 規則

### 1. convert-droid
- state file: `output_dir/.convert_state.json`
- 重新執行相同命令時，會從已處理 episode 後面繼續
- 若 `manifest.json` 已完成且 state 也標成完成，則會直接返回已完成 manifest

### 2. train-sharded
- checkpoint: `/workspace/checkpoints/droid_*_train_ckpt.pkl`
- 每個 shard 預設存一次 checkpoint
- 重新執行相同訓練腳本時，會從 `next_epoch / next_shard_idx` 繼續

### 3. fine-tune-fino
- checkpoint: `/workspace/checkpoints/fino_finetune_ckpt.pkl`
- 每個 shard 預設存一次 checkpoint
- 重跑相同腳本會延續 horizon head adaptation


### 如果 Step 04 因 RunPod 中斷留下壞掉的 trailing shard

新版 converter 會自動偵測 `output_dir/train|calib|eval` 內最後一個損壞的 trailing shard，並將其改名為 `*.corrupt.<timestamp>`，然後從最後一個可讀取的 shard 之後繼續。

因此，**不要手動刪整個 `droid_curated` 目錄**；直接重跑同一支 `04_convert_droid_curated.sh` 即可。

## 九、剩餘時間怎麼看

所有長流程都會印：

- processed / total
- elapsed
- rate
- remaining

例如：

```text
[convert-droid] 89696/95658 episodes | elapsed 09:43:52 | rate 2.56/episode/s | remaining 38:49
```

如果你看到這種格式，就表示：
- **還沒跑完**
- remaining 是依目前平均速度估算

## 十、Step 04 為什麼現在也會用到 GPU

第二章 2.4–2.5 強調的是 **frozen visual encoder + event-driven verification**。本版 `convert-droid` 新增：

- `--precompute-encoder`
- `--precompute-device`

RunPod 腳本預設會把 Step 04 的轉換做成：

- TFDS / RLDS 讀取仍由 CPU / I/O 主導
- 但若 `USE_HF=1`、`ENCODER=siglip2_dinov2`、`EDSVFH_DEVICE=cuda`，則會在轉換時用 GPU **預計算 frozen visual/context features**，並寫入 shard

這樣：
- 更符合第二章的兩階段策略
- 後續 `train-sharded` 不必再重複抽同一批特徵

### RunPod 中途中斷與壞掉 shard 的續跑

`convert-droid` 現在會在 resume 時自動檢查各 split 的最後一個 shard。若 RunPod 斷線或 Pod 被中止導致最後一個 HDF5 shard 壞掉，系統會自動將其改名為 `*.corrupt.<timestamp>` 並從最後一個可讀 shard 後續跑；不需要手動刪掉整個 `droid_curated` 目錄。若損壞的 shard 不是 trailing shard，程式仍會停止並要求人工檢查，以避免靜默遺失資料。


## RunPod 環境注意事項

- 建議使用 **Runpod PyTorch 2.8.0** 官方模板。
- **不要手動執行** `pip install --upgrade transformers torch torchvision`。這會破壞模板內已驗證的 PyTorch/CUDA 相容組合。
- 若要啟用第二章正式 encoder（`siglip2_dinov2`），請執行 `bash scripts/runpod/00_repair_hf_runtime.sh`；該腳本只會安裝 `transformers/safetensors/huggingface-hub`，**不會升級 torch/torchvision**。
- 若在 Blackwell / RTX PRO 6000 WK 上遇到 cuDNN 初始化問題，預設會啟用 `EDSVFH_DISABLE_CUDNN=1` 以維持 Step 04 的 GPU frozen-feature 預計算穩定性。


## 十、若要整個重跑一次（保留 raw 下載資料）

先執行：

```bash
bash scripts/runpod/18_reset_experiment_state_once.sh --yes
```

這個指令只會刪除一次歷史遺留的：
- converted shards
- checkpoints
- logs
- 主要 bundle outputs

它**不會**自動在未來每次訓練前執行，也**不會**刪除 `data/raw`。


### v17 performance/runtime notes

- Step 04 now uses **batched** frozen-feature extraction on GPU. The batch size is controlled by `EDSVFH_CONVERT_BATCH_SIZE` (default `16`).
- `EDSVFH_DISABLE_CUDNN` now defaults to `0`; the encoder only falls back to disabling cuDNN after a real runtime failure.
- Do **not** manually run `pip install --upgrade torch torchvision`. Use `scripts/runpod/00_repair_hf_runtime.sh` for Hugging Face dependencies only, and use `scripts/runpod/00_force_repair_torch_stack.sh cu124|cu128` only if the torch stack has already been polluted.


### v17 throughput note

The slower-than-v11 behavior came from per-frame SigLIP2+DINOv2 precomputation in Step 04. v17 switches this to **batched GPU precomputation** (`CONVERT_BATCH_SIZE`, default `16`) while keeping TensorFlow off the GPU and cuDNN enabled by default. This preserves the Chapter-2 design while restoring practical conversion speed.
